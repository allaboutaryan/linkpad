# Justification

This prompt is strong because it gives a clear product goal first, then narrows the work into exact technical expectations. It explains who LinkPad is for, what the user flow should be, and which features should be included without making the task vague. The completeness is covered by naming the frontend, backend, realtime layer, Electron wrapper, endpoints, room behavior, and documentation requirements. It also keeps adherence tight by saying what not to build, such as login, databases, dashboards, and unrelated scope. The efficiency requirement is natural because the app only needs temporary rooms and in-memory sync, so the model is guided toward a simple working design. Repetitiveness is handled by asking for clean helpers and avoiding duplicate room logic. The tone requirement keeps the final product readable and human instead of sounding like a generic template. Accuracy is protected by requiring commands, ports, event names, dependencies, and file paths to match the actual project. Overall, the prompt gives enough detail for a full codebase while still leaving room for clean engineering decisions.
